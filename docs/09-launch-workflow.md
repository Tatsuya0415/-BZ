# 実行ワークフロー — 立ち上げ実務手順(2026年7月14日起点)

本ドキュメントは「計画」から「実行」への移行に伴い、実際にサービスを開始するための
具体的なタスク・スケジュール・週次オペレーションを定義する。副業(本業と並行、週5〜10時間程度)を
前提とし、Claude Codeを開発・コンテンツ制作の実務ツールとして活用する想定で設計する。

## 0. 前提条件

- 起点日: 2026-07-14(本ドキュメント作成日)
- 運営形態: 個人事業主としてスタート(法人化はフェーズ3以降で検討)
- 目標: **12週間でMVP(診断+セルフ登記ガイドの主要機能)を公開し、フェーズ1(集客優先)の定常運用に入る**
- MVPの範囲は[03-business-model.md](03-business-model.md)フェーズ1の方針(収益化しない・無料ツールに全振り)に準拠

## 1. 全体スケジュール(俯瞰)

| 週 | フェーズ | 主なアウトプット |
|---|---|---|
| W0-1 | 事業準備 | 開業届・ドメイン・法務ひな形の記入完了 |
| W1-2 | 技術選定・環境構築 | 開発環境、サイト骨格 |

| W9以降 | 定常運用 | 日次/週次ルーティンに移行、KPIモニタリング開始 |

## 3. フェーズ1: 技術選定・開発ワークフロー(W1-6)

### 技術スタック(提案)

| 領域 | 提案 | 理由 |
|---|---|---|
| フロントエンド/フルスタック | Next.js(App Router) | Vercelとの親和性、SSRでSEOに強い |
| ホスティング | Vercel | [04-cost-estimate.md](04-cost-estimate.md)の無料枠前提と整合 |
| 書式PDF生成 | `@react-pdf/renderer` または HTML→PDF(Puppeteer) | 相続関係説明図・遺産分割協議書・登記申請書のひな形出力 |
| 進捗保存 | Supabase(無料枠)または簡易的にlocalStorage+エクスポート機能 | ログイン不要設計との両立を優先 |
| アクセス解析 | GA4 | 無料([04-cost-estimate.md](04-cost-estimate.md)) |
| 開発方法 | Claude Code(このセッション)で実装・レビューを継続 | 副業の限られた時間を圧縮 |

### 実装優先順位(MoSCoW、[05-site-design-requirements.md](05-site-design-requirements.md)の機能要件に対応)

- **Must**: 診断ロジック(ルールベース)、質問フロー画面、書式ひな形自動生成、必要書類チェックリスト、
  全画面免責表示、特商法・利用規約・プライバシーポリシーページ
- **Should**: 進捗管理(保存・再開)、専門家送客導線(問い合わせフォーム)、SNSシェアボタン
- **Could**: AIチャット相談、コラム機能の高度化(タグ・関連記事)
- **Won't(フェーズ1では実装しない)**: 決済機能、有料プラン、複雑な会員機能

### 週次開発タスク分解

| 週 | タスク |
|---|---|
| W1-2 | Next.jsプロジェクト作成、デザイントークン([06-component-spec.md](06-component-spec.md))反映、共通コンポーネント(Button/Input/Card/Badge/Stepper)実装 |
| W2-3 | 診断質問フロー画面 + ルールベース診断ロジック実装 |
| W3-4 | 書式ひな形自動生成(相続関係説明図・遺産分割協議書・登記申請書のPDF出力) |
| W4-5 | 必要書類チェックリスト、進捗管理、専門家送客フォーム |
| W5 | 法務レビュー結果の反映(指摘事項の修正) |
| W6 | 免責表示の全画面適用、特商法・規約ページ実装、QA・レスポンシブ確認 |

### 開発サイクル(週次ルーティン)

1. 週初め: 今週のタスクを1〜3個に絞り込む(副業前提でスコープを守る)
2. Claude Codeで実装(`/code-review`スキルで自己レビューしてからマージ)
3. ローカル/プレビュー環境で動作確認(スマホ・PC両方)
4. 週末にVercelへデプロイ

## 4. フェーズ2: 法務レビューワークフロー(W5、公開前必須)

[08-legal/legal-risks.md](08-legal/legal-risks.md)の「推奨アクション」に基づく具体手順:

1. **準備資料の作成**: 機能一覧(診断ロジックの分岐例、書式自動生成の対象書類、AIチャットの有無)、
   画面フロー図、免責表示文言案を1つの資料にまとめる
2. **依頼先候補**: 弁護士ドットコム・ココナラ法律相談等のスポット相談サービス、または地域の
   司法書士会・税理士会の相談窓口(スポット5〜20万円、[04-cost-estimate.md](04-cost-estimate.md)参照)
3. **確認項目**:
   - 非司法書士行為・非税理士行為・非弁行為への該非
   - 個人情報保護法上の取扱い(利用目的の特定・第三者提供の同意取得方法)
   - 特定商取引法表記の要否(フェーズ1は無料のため原則不要だが、将来の有料化を見据え確認)
4. **反映**: 指摘事項をリスト化し、W6のQA工程で機能・文言を修正する
5. **記録**: レビュー実施日・指摘内容・対応内容を残しておく(将来の追加専門家レビュー時の参考にする)

### 公開前最終チェックリスト

- [ ] 特定商取引法に基づく表記・利用規約・プライバシーポリシーへの導線がフッターにある

### 実施手順

1. W7: 身内・知人5〜10名程度に先行公開し、フィードバックを収集(使いやすさ、診断結果の納得感等)
2. W7後半: フィードバックを反映(致命的でない指摘は正式公開後に回すことも可)
3. W8: 正式公開。SNS・(準備していれば)知人経由での告知投稿

## 9. KPIモニタリングワークフロー

[03-business-model.md](03-business-model.md)のフェーズ1 KPIを実務指標に落とす:

| KPI | 計測方法 | 頻度 | フェーズ2検討の目安 |
|---|---|---|---|
| 診断完了率 | GA4イベント(診断開始→完了) | 週次 | 継続的に50%超で安定 |
| セルフガイド利用率 | 書式ダウンロード数/診断完了数 | 週次 | - |
| SNS経由流入・シェア数 | GA4流入元、シェアボタンクリック数 | 週次 | - |
| リピート率 | 進捗管理への再訪率 | 月次 | - |
| 月間診断完了数(絶対数) | GA4 | 月次 | 目安として月500件前後で送客手数料モデルの検証開始を検討 |
| 指名検索数 | Search Console(サービス名検索) | 月次 | 継続増加傾向 |

閾値は初期仮説であり、実データが蓄積され次第、[03-business-model.md](03-business-model.md)の
フェーズ2移行基準として更新すること。

## 10. 提携専門家開拓ワークフロー(フェーズ2準備、種まきはW9以降に並行開始)

1. 開拓リスト作成: 地域の若手司法書士・税理士事務所、相続専門を掲げる事務所を中心にリストアップ
2. アプローチ: 「着手金なし・成約時マージンのみ」の送客提案を伝えるメール/問い合わせフォームでの打診
3. 提案資料: サービス概要・想定送客数・紹介条件(手数料率は個別交渉)をまとめた1ページ資料を用意
4. 契約: 送客契約書は必ず専門家(弁護士)確認を経て締結する([08-legal/legal-risks.md](08-legal/legal-risks.md)参照)
5. この段階では「提携準備」に留め、正式な送客手数料モデルの稼働はフェーズ2のKPI基準達成後とする

## 11. リスク管理・エスカレーションフロー

- **新機能追加時**: 非弁・非司法書士・非税理士行為に該当しないか、追加前に必ずセルフチェック
  ([08-legal/legal-risks.md](08-legal/legal-risks.md))→ 疑義がある場合は専門家に確認してから実装する
- **ユーザーからの個別相談的な問い合わせ**: 一般的情報提供の範囲を超える質問には「個別の法律相談は
  提携専門家をご案内します」という定型回答でエスカレーションする
- **クレーム・トラブル**: 事実確認→ 免責事項の範囲内かどうかを確認→ 必要に応じて専門家に相談→
  利用規約に基づき対応する

## 12. まとめ

本ワークフローは12週間でのMVP公開を軸にしているが、副業運営のため遅延は許容する。
重要なのは**W5の専門家法務レビューを飛ばさないこと**と、**公開後は日次/週次ルーティンに移行し、
KPIを見ながらフェーズ2移行タイミングを判断すること**の2点である。
