# AIエージェント体制構築 — 実質1名運営を支えるAIチーム設計

個人(創業者1名)が本業と並行してサービスを回すため、役割ごとに特化したAIエージェントを
「仮想チーム」として配置し、創業者は**最終判断・対外的確定・法務ゲートキーパー**に専念する体制を作る。
本ドキュメントは、その体制の設計と、実際にClaude Code上で運用するための具体的な仕組みを定義する。

## 1. 基本方針

1. **AIは下書き・一次処理・監視・提案までを担当し、最終判断と対外的な確定行為は必ず人間(創業者)が行う**
2. 特にYMYL領域([07-marketing-plan.md](07-marketing-plan.md))・法的リスク領域
   ([08-legal/legal-risks.md](08-legal/legal-risks.md))に関わるアウトプットは、AIが生成した時点では
   「未確定のドラフト」として扱い、人間のファクトチェック・承認を経るまで公開・送信しない
3. エージェントは役割ごとに固定の指示テンプレート(システムプロンプト相当)を持ち、都度書き直さない
4. 体制は事業フェーズに応じて拡張する(フェーズ1は最小構成、フェーズ2以降で拡張)

## 2. 体制図(フェーズ1: MVP公開〜集客優先期)

```
創業者(意思決定・最終承認・専門家対応・法務ゲートキーパー)
│
├─ ① コンテンツエージェント群
│   ├─ SEOリサーチ・構成エージェント
│   ├─ 記事ドラフト執筆エージェント
│   └─ [人間ゲート] ファクトチェック・公開承認 ← 必須
│
├─ ② SNS運用エージェント群
│   ├─ 投稿ドラフト生成エージェント(X/Instagram/TikTok台本)
│   ├─ 投稿スケジューリング補助
│   └─ [人間ゲート] トーン確認・投稿ボタンは人間が押す
│
├─ ③ プロダクト開発エージェント群
│   ├─ 実装エージェント(Claude Code)
│   ├─ コードレビュー・QAエージェント(/code-review, /security-review)
│   └─ [人間ゲート] 動作確認・デプロイ承認
│
├─ ④ カスタマー対応エージェント
│   ├─ FAQ一次回答エージェント(診断チャット、フェーズ2以降)
│   └─ [人間ゲート] 個別法律相談化した場合のエスカレーション判定
│
├─ ⑤ モニタリング・分析エージェント群
│   ├─ KPI週次レポーティングエージェント
│   ├─ 競合・法改正モニタリングエージェント
│   └─ [人間] 戦略判断(フェーズ移行の意思決定)
│
└─ ⑥ 法務ゲートキーパー(人間+外部専門家、AI代替不可)
```

## 3. 各エージェントの定義

| # | エージェント | 役割 | インプット | アウトプット | 頻度 | 人間レビューポイント |
|---|---|---|---|---|---|---|
| 1-1 | SEOリサーチ・構成 | キーワード調査、記事構成案の作成 | ターゲットキーワード、検索意図 | 記事構成案(見出し・要点) | 週次 | 構成の妥当性確認 |
| 1-2 | 記事ドラフト執筆 | 構成案から本文ドラフト生成 | 構成案 | 本文ドラフト | 記事ごと | **法令根拠・数値の一次情報照合必須**(YMYL) |
| 2-1 | SNS投稿ドラフト生成 | テーマから投稿文案・台本を生成 | 週次テーマ、過去の反応データ | 投稿文案(複数案) | 週次バッチ | 事実確認・トーン確認後に投稿 |
| 2-2 | 投稿スケジューリング | 承認済み投稿の予約管理 | 承認済み投稿文 | 予約投稿 | 週次 | 予約前の最終確認 |
| 3-1 | 実装エージェント | 機能要件からコード実装 | [09-launch-workflow.md](09-launch-workflow.md)のタスク | 実装コード・PR | 都度 | 動作確認、デプロイは人間が実行 |
| 3-2 | コードレビュー・QA | 差分のバグ・改善点指摘 | コード差分 | レビュー指摘リスト | 都度 | 重大指摘の対応要否は人間判断 |
| 4-1 | FAQ一次回答 | 一般的な制度説明への回答生成 | ユーザー質問 | 回答案(一般的情報提供に限定) | リアルタイム | 個別相談化の兆候を検知したら人間対応に切替 |
| 5-1 | KPIレポーティング | アクセス解析データの週次サマリ化 | GA4/Search Consoleデータ | 週次サマリレポート | 週次 | 異常値・トレンドの解釈は人間が行う |
| 5-2 | 競合・法改正モニタリング | 競合サイト・官公庁発表の変化点検知 | Web検索結果 | 変化点サマリ | 月次 | 対応要否・優先度は人間判断 |
| 6 | 法務ゲートキーパー | 新機能の非弁・非司法書士・非税理士該当性チェック | 機能追加案 | 該非判断・専門家確認要否 | 新機能追加のたび | **AIに代替させない。必要に応じ外部専門家に確認** |

## 4. ガバナンスルール(AIに任せてよいこと/いけないこと)

| 業務 | AIに任せてよい範囲 | 必ず人間が行うこと |
|---|---|---|
| コラム記事 | 構成案・ドラフト生成 | 法令根拠のファクトチェック、公開ボタン |
| SNS投稿 | 文案・台本のドラフト生成 | 最終トーン確認、投稿の実行 |
| 診断ロジック | テストケース生成、分岐網羅性のチェック補助 | ロジック設計・診断基準の決定(法的リスクに直結するため) |
| コード実装 | 実装・リファクタリング・テスト作成 | レビュー承認・本番デプロイ |
| ユーザー対応 | 一般的なFAQ回答のドラフト | 個別具体的な法律・税務相談への回答(提携専門家へ誘導) |
| 専門家紹介 | 紹介先候補の整理・提案資料ドラフト | 紹介の可否判断、契約締結 |
| 法的文書 | ひな形のドラフト作成(本リポジトリの[08-legal/](08-legal/)がこれに該当) | 専門家レビューを経た最終確定 |

この表は[08-legal/legal-risks.md](08-legal/legal-risks.md)の回避設計方針(「代行ではなくセルフヘルプ
支援に徹する」)を、AI運用面にも一貫して適用するためのものである。

## 5. Claude Code環境での具体的な実装方法

このセッション自体がClaude Code上で動いているため、上記のエージェント体制の一部は既存の仕組みで
そのまま運用できる。

| エージェント | 具体的な実現方法 |
|---|---|
| 実装エージェント | このリポジトリでの通常の開発依頼(Claude Codeへの実装指示) |
| コードレビュー・QA | `/code-review`スキル(コミット前の差分レビュー)、`/security-review`スキル |
| 動作確認 | `/verify`スキル(変更が実際に動くかをエンドツーエンドで確認) |
| SEOリサーチ・構成、記事ドラフト | 通常の会話でのAI活用(記事構成・ドラフト依頼) |
| KPI週次レポーティング | GA4データを人間が共有し、週次でサマリ化を依頼する運用、または後述のRoutine化 |
| 競合・法改正モニタリング | Web検索を伴う定期的な調査依頼 |
| 定期実行の自動化 | `create_trigger`(Routine)機能で「毎週月曜朝にSNS投稿ドラフトを作成」等を
  スケジュール可能。ただし**投稿の最終実行は人間が承認**する運用とする |

### Routine化の提案(要確認)

以下のような定期タスクはRoutineとして自動化できる。実際に設定するかどうかはユーザーの判断に委ねる:

- 「毎週月曜9時: 今週のSEO記事構成案とSNS投稿ドラフトを作成して提示する」
- 「毎週金曜: その週のKPI確認事項をリマインドする」

いずれも**生成されたドラフトを人間が確認・承認するまでは公開・送信しない**という第4章のガバナンス
ルールと矛盾しない範囲での自動化に限定する。設定を希望する場合は、頻度・内容を指定して依頼すること。

## 6. エージェント運用の指示テンプレート原則

各エージェントへの依頼時は、以下を指示に含めることを徹底する(特に1-2, 2-1, 4-1):

1. 「本サービスはYMYL領域(相続・法律・税務)を扱うため、断定的な個別助言をしない」
2. 「一般的な情報提供に留め、個別ケースへの適用は『提携専門家にご相談ください』という形で誘導する」
3. 「生成物は下書きであり、公開・送信前に人間のファクトチェック・承認が必須である」

これらは[08-legal/legal-risks.md](08-legal/legal-risks.md)の回避設計方針をAIへの指示レベルまで
落とし込んだものであり、コンテンツ・SNS・カスタマー対応の全エージェントに共通適用する。

## 7. フェーズ2以降の拡張

| フェーズ | 追加するエージェント | 目的 |
|---|---|---|
| フェーズ2 | 専門家マッチング補助エージェント | 診断結果・案件属性に応じた送客先候補の提案(最終判断は人間) |
| フェーズ2 | 提携専門家向け一次対応エージェント | 提携先からの問い合わせへの一次回答ドラフト作成 |
| フェーズ3 | データ分析・レポート生成エージェント | 蓄積した診断データを基にした相続傾向レポートの作成補助(メディアとしての権威性強化、[03-business-model.md](03-business-model.md)フェーズ3参照) |

## 8. まとめ

この体制の核心は「**AIに速度と物量を任せ、人間は法的リスクのゲートと最終承認に集中する**」という
役割分担である。特に記事・SNS・FAQの3領域はAI生成→人間承認のワンクッションを絶対に省略しないこと。
これにより、副業という時間制約の中でも、コンテンツ量産とSNS運用を継続しながら、
[08-legal/legal-risks.md](08-legal/legal-risks.md)で定義した法的リスクを回避する体制を両立させる。
