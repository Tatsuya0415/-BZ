# サイト設計案・要件定義

## 1. サイトマップ

```
TOP
├── 状況診断(セルフ可否判定)
│   ├── 質問フロー(Stepperで進捗表示、1画面数問)
│   └── 診断結果(セルフ推奨 / 代行推奨 / ハイブリッド)
├── セルフ登記ガイド
│   ├── 必要書類チェックリスト
│   ├── 書式ひな形作成(相続関係説明図・遺産分割協議書・登記申請書)
│   └── 進捗管理(マイページ的な保存機能)
├── セルフ相続税申告シミュレーター
│   ├── 財産構成の簡易入力
│   └── 「自分で申告できる可能性」の目安表示 + 詳細への誘導
├── 代行専門家紹介(複雑案件向けCTA)
│   └── 提携司法書士・税理士への問い合わせ導線
├── コラム(SEO)
│   └── 「相続登記 義務化 期限」等のロングテールキーワード記事
├── 会社概要
├── 特定商取引法に基づく表記
├── プライバシーポリシー
└── 利用規約
```

## 2. 診断フロー(コア体験)

1. **TOP** → 「まずは30秒で診断」CTA(CTA Block)
2. **質問画面**(1画面1〜2問、Stepperで進捗表示、選択式はCard/Buttonで回答)
   - 例: 相続財産に不動産は含まれるか / 相続人は何人か / 相続人間で意見の相違はあるか /
     海外在住の相続人がいるか / 財産総額のおおよそのレンジ
3. **診断結果画面**
   - **Summary Panel**: 「あなたはセルフ登記向きです」等の一言サマリ + 推奨理由
   - **Badge**: 該当する手続き種別タグ(相続登記/相続税申告/相続放棄 等)を複数表示
   - **Accordion**: 手続きごとの詳細説明・必要書類を段階開示(初見のユーザーを情報過多にしない)
   - **CTA Block**: 診断結果に応じて出し分け
     - セルフ推奨 → 「セルフ登記ガイドへ進む」
     - 代行推奨 → 「提携専門家に相談する」
     - ハイブリッド → 両方の導線を提示、優先度を明示

## 3. 機能要件

| 機能 | 概要 | 優先度 |
|---|---|---|
| 診断ロジック | 質問への回答から「セルフ/代行/ハイブリッド」を判定するルールベースのロジック(初期はAIではなくルールベースを推奨。理由は[コスト試算](04-cost-estimate.md)と[法的リスク](08-legal/legal-risks.md)を参照) | 必須 |
| 書式自動生成 | 入力内容から相続関係説明図・遺産分割協議書ひな形・登記申請書ひな形をPDF/Wordで出力 | 必須 |
| 進捗管理 | ログイン不要 or 簡易アカウントで、診断結果・入力内容を保存し途中から再開できる | 高 |
| 専門家送客導線 | 代行が必要なケースで、提携司法書士・税理士への問い合わせフォームへ誘導 | 高 |
| AIチャット相談 | よくある質問への一般的な回答(個別の法律相談は行わない旨を明示) | 中(フェーズ2以降でも可) |
| SNSシェア機能 | 診断結果画面から「診断結果をシェア」できるボタン(バイラル集客の起点) | 高 |
| SEOコラム | 検索流入獲得のための記事一覧・記事詳細ページ | 高 |

## 4. 非機能要件

- **個人情報保護**: 診断・書式作成の過程で相続人の氏名・住所・財産情報など機微な個人情報を扱うため、
  通信のTLS化、保存データの暗号化、不要なデータの非保持(進捗保存を使わないユーザーの入力はセッション限り
  で破棄する等)を設計段階で組み込む
- **法令順守**: 診断結果・書式は「法的助言ではない」旨を全画面で明示し、有償の書類作成代理・申告代理に
  該当しない設計であることを常に確認する([legal-risks.md](08-legal/legal-risks.md)参照)
- **レスポンシブ対応**: 主要ターゲット(相続登記義務化を意識し始めた40〜60代)はPC・スマホ両方からの
  アクセスが見込まれるため、両対応必須
- **可用性**: 副業規模のため過度な高可用性investment は不要。Vercel等のマネージドホスティングで十分

## 5. ターゲットユーザー像

- 相続登記義務化のニュースを見て「自分はどうすればいいか分からない」と感じている40〜60代
- 相続財産がシンプル(自宅の土地建物・預貯金程度)で、税理士に頼むほどではないと感じている層
- 士業への相談に心理的・費用的ハードルを感じ、まず自分で調べたい層
